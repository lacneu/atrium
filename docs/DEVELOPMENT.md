# Development Guide

This is the local workflow for hacking on Atrium. The repo has two
runnable parts: the **app** (React front end + Convex backend, at the repo root)
and the **bridge** (`bridge/`). You only need the bridge running when you want to
exercise a real OpenClaw gateway end to end.

## Prerequisites

- Node.js (see the engines in `package.json`; Node 24 is what the Docker build
  uses).
- An OpenClaw gateway only if you want a live end-to-end loop. The app and its
  tests run fine without one.

## App: install and run

```bash
npm install
```

The fastest way to get a local stack up is the dev script:

```bash
./dev.sh
```

It starts a local anonymous Convex backend and the Vite dev server, then prints
the URL to open. Logs are written under `/tmp`. Stop it by re-running the script
(it tears down its own previous instances first) or by stopping the processes.

To run the two pieces yourself instead:

```bash
npx convex dev      # local Convex backend (also: npm run dev:convex)
npm run dev         # Vite dev server (front end)
```

In local dev there is no `/config.json`, so the front end reads the Convex URL
from `VITE_CONVEX_URL` (set in `.env.local`). `npx convex dev` will create/update
that for you.

## Tests and quality gates

```bash
npm run typecheck   # paraglide compile + tsc --noEmit
npm test            # paraglide compile + i18n gates + vitest run
npm run build       # vite build
```

Convex functions are tested with `convex-test` through Vitest. `npm test` also
runs the two i18n gates:

- **parity** — every message key in the base locale must exist in every other
  locale, or the build fails.
- **literal ratchet** — guards against new hard-coded accented UI strings that
  bypass the message catalog; the allowed count only ratchets down.

### Internationalization

UI strings live in the inlang message catalog and are compiled by Paraglide into
`src/paraglide/` (git-ignored, regenerated by `npm run paraglide:compile`, which
the `test`/`typecheck` scripts run for you). Add a new string to the catalog and
reference it as a typed `m.key()` call rather than hard-coding it.

## Bridge

The bridge is a self-contained Node/TypeScript worker with its own `package.json`
and tests in `bridge/`:

```bash
cd bridge
npm ci
cp .env.example .env       # fill OPENCLAW_TOKEN, OPENCLAW_DEVICE_IDENTITY, …
npm run typecheck          # tsc --noEmit
npm test                   # vitest: normalizer, multiplex, media-fetcher, …
npm run build
npm start                  # runs dist/index.js with --env-file=.env
```

`bridge/.env.example` documents every variable, including the Convex ingest URL
and the shared secrets. Secrets live only in the bridge environment — never
commit `.env`.

### Live bridge ↔ gateway loop

`bridge/local-openclaw/` is a version-pinned OpenClaw harness for exercising the
bridge↔gateway seam locally. `bridge/local-openclaw/up.sh` boots a local gateway
and prints the overrides to run the bridge against it. A full end-to-end test
also needs a reachable Convex (run the app's `npx convex dev`). See
[OPENCLAW_VERSION_COMPAT.md](OPENCLAW_VERSION_COMPAT.md) for the per-version
replay checklist.

## Frame capture and regression testing

The bridge's correctness depends on real OpenClaw frame shapes, which differ by
gateway version and carry hard-won edge cases. The regression suite is built from
**captured real frames**, not invented ones: when a regression is found, add the
failing capture as a fixture first, then extend the normalizer until it passes —
the bridge↔Convex contract stays stable.

## Adding a new front end

A third-party front end should subscribe to Convex (the reactive source of
truth), or, if it needs to talk to the bridge directly, depend on the
**normalized** contract in [BRIDGE_PROTOCOL.md](BRIDGE_PROTOCOL.md) rather than on
raw gateway frames. New normalized fields are added to the bridge protocol with a
test first.
